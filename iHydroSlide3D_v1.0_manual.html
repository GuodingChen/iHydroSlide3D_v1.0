iHydroSlide3D v1.0: Coupled hydrological model and three-dimensional landslide model

Version: 1.0 - 20210210 (10 February 2021)

 

1. iHydroSlide3D contains three core modules:

  • Hydromodel
  • 3D landslide model
  • Soil moisture downscaling method

2. Necessary data within folders

3. Detailed descriptions for files

The model features the feasibility of detached simulating resolution applied for submodules, by embedding a soil moisture downscaling method. Here we adopt adjectives "coarse" and "fine" denoting the resolution applied for hydrological module and landslide module, respectively.

  • DEM

    Digital elevation model with coarse resolution.

  • FAC

    Flow accumulation with coarse resolution.

  • FDR

    Flow direction with coarse resolution.

  • LANDCOVER

    Land surface cover with coarse resolution.

  • MASK

    A matrix with binary value (0/NAN and 1) to delineate the objective river basin that serves as a extraction map.

  • SOIL

    Soil texture map with coarse resolution and is marked with USDA soil code from 1 to 12.

  • aspect_fine

    Geographical aspect angle (in degree) with fine resolution.

  • DEM_fine

    digital elevation model with fine resolution.

  • mask_fine

    mask map with fine resolution.

  • slope_fine

    Geographical slope angle (in degree) with fine resolution.

  • aspect_coarse

    Geographical aspect angle (in degree) with coarse resolution.

  • curvature_coarse

    Curvature with coarse resolution.

  • curvature_fine

    Curvature with fine resolution.

  • TWI_coarse

    Topographic wetness index with coarse resolution.

  • TWI_fine

    Topographic wetness index with fine resolution.

  • Precipitation data and Potential ET data

    Matrix format with coarse resolution.

  • COEM

    Overland runoff velocity coefficient.

  • FC

    Saturated hydraulic conductivity.

  • IM

    Percentage impervious area.

  • WM

    Available water capacity.

Notes: iHydroSlide3D v1.0 currently supports several different options for file formats (with ASCII, TIFF, and TXT) and map projections, of which the Geographic Tagged Image File Format (GeoTIFF) is preferred and recommended for its distinct advantage of including native compression capabilities, making the file sizes smaller.

4. Parameters

Prior information on parameters is necessary for this model and need to be handled with the utmost care. As a matter of fact, most of the parameters are determined by available datasets and field records, while few of them are calibrated manually based on computational experimental tests. For the quick star for users, some ranges/ballpark figures are also provided here. Some of them are distributed over the modelling region; however they will be set as default values if no corresponding inputs can be read. Users could of course change the default values by recoding them in specific code segments (still need to take value in range).

  • TimeStep: Time step of the simulation.

    Default value: NA

    Range: NA

  • W0: Initial Value of Soil Moisture; unit is mm

    Default value: NA

    Range: NA

  • R: Initial channel streamflow.

    Default value: NA

    Range: NA

  • RS: Initial unit overland streamflow.

    Default value: NA

    Range: NA

  • SI0: Initial value of Interflow reservoir.

    Default value: NA

    Range: NA

  • RainFact: The multiplier on the precipitation field.

    Default value: 1

    Range: [0.5,1.2]

  • Ksat: The Soil saturate hydraulic conductivity; unit is mm/day

    Default value: 120

    Range: [0,1000]

  • WM: The Mean Water Capacity; unit is mm

    Default value: 120

    Range: [80,200]

  • B: The exponent of the variable infiltration curve.

    Default value: 0.25

    Range: [0.05,1.5]

  • ISA: Impervious area ratio.

    Default value: 0.25

    Range: [0,0.2]

  • KE: The factor to convert the PET to local actual.

    Default value: 0.95

    Range: [0.1,1.5]

  • coeM: Overland runoff velocity coefficient.

    Default value: 90

    Range: [1,150]

  • Landslide_density: Density of the random ellipsoid over the area.

    Default value: 100

    Range: [50,+∞]

    Note: users should set this value with utmost care. Model runtime will increase significantly as this value increases.

  • Divide_tile_number: Number of divisions of study area.

    Default value: 40

    Range: [20,+∞]

    Note: different tile number will not affect the final outputs but change the way to merge them. It's better for the user to determine this value based on "Ncores". gIt’s a good choice for users to set it divisible by “Ncores”, to make better use of computing resources.

  • min_ae: The minimum length of a random ellipsoid.

    Default value: NA

    Range: NA

  • max_ae: The maximum length of a random ellipsoid.

    Default value: NA

    Range: NA

  • min_be: The minimum width of a random ellipsoid.

    Default value: NA

    Range: NA

  • max_be: The maximum width of a random ellipsoid.

    Default value: NA

    Range: NA

  • UserDefinedNcores: Number of parallel computational cores.

    We provide a option to identify the max number of cores in local environment, if the users state "yes" in "ControlFile".

    Note: for personal computers (PC, usually 8 cores and 16GB of RAM), we recommend the value ≤ 5, considering the limited RAM. If the error says "out of memory", decrease the value until the model can run. For users who have access to high performance computing (HPC), this value can be flexibly customized. We consider the latter a appropriate way to perform the iHydroSlide3D v1.0.

  • A: A parameter depending on the kind of soil and is linked to the peak shear stress at failure.

    Default value: NA

    Range: NA

  • λ: A numerical parameter which allows to translate the peak of the curve along the abscissa axis.

    Default value: NA

    Range: NA

5. Software and hardware requirements

iHydroSlide3D v1.0 is written in MATLAB language and capable of running in a parallel manner, currently supporting Linux and Windows operating systems. Users are recommended to install the MATLAB platform with versions of R2018B and above in case that some unexpected errors will emerge due to the unrecognized functions. The code can currently run successfully on Windows 10 and CentOS release 6.6, but is expected to work in other LINUX and Windows environments, too. The computing core needs to be more than or equal to 2 in a tested machine to ensure the computational efficiency. The best option is to submit the code to computing clusters or cloud computing services, however, PCs can also be attempted if the users could take their time.

Given that the model is performed in the MATLAB environment, several toolboxes (e.g., Mapping Toolbox, Parallel Computing Toolbox, Curve Fitting Toolbox) are necessary to ensure the model works. In addition to the most basic installation content, the software requires additional toolboxes: Curve Fitting Toolbox, Global Optimization Toolbox, Optimization Toolbox, Partial Differential Equation Toolbox, Statistics and Machine Learning Toolbox, Symbolic Math Toolbox, MATLAB in the Cloud, MATLAB Parallel Server, Parallel Computing Toolbox. For users who have most of the toolbox installed by default, they can run the software easily. If users have a limited installation of MATLAB, they are advised to check in advance. Users can also directly run the code without concerns and install the toolboxes needed according to the error prompt. If you have tried many ways and it still doesn't work, please don't hesitate to contact us.

6. Run the code step by step

  1. Prepare the necessary data

    All the data mentioned above should be acquired and preprocessed before the code being executed. The data must be correctly placed in the specified folders.

  2. Set the "ControlFile"

    Users may find the example of the case, for Windows:

    xxxxxxxxxx
    ControlFile.ProjectWindows

    and for Linux, please find:

    xxxxxxxxxx
    ControlFile.ProjectLinux

    Such file is so important that should be treated by users with utmost care. Users are required to set the simulation period by modifying "start time" and "end time". We note that there are two simulation durations including the main- and sub-durations. The former denotes the simulation of the hydrological processes, while the later controls the landslide model outputs. For instance, they can be set as:

    xxxxxxxxxx
    StartYear        =  2012 # in the format of yyyy
    StartMonth       =    06 # in the format of mm
    StartDay         =    05 # in the format of dd
    StartHour        =    03 # in the format of hh
    StartMinute      =    00 # in the format of uu
    StartSecond      =    00 # in the format of ss
    EndYear        =    2012 # in the format of yyyy
    EndMonth       =      06 # in the format of mm
    EndDay         =      05 # in the format of dd
    EndHour        =      14 # in the format of hh
    EndMinute      =      00 # in the format of uu
    EndSecond      =      00 # in the format of ss

    and for landslide model output, we can set:

    xxxxxxxxxx
    LandslideOutStart_data = 2012-06-05 03:00:00
    LandslideOutEnd_data = 2012-06-05 14:00:00

    The file contains all parameters above that need to be assigned with specific value. Part of them are provided a option to be read in a distributed manner, which can be easily modified by users by choosing "Uniform" and "Distributed" pattern. In this case, all the data is associated with the path and format of the file and, for example, is set as:

    xxxxxxxxxx
    ParamFormat      =  tif
    ParamPath        =  ".\Parameters\"

    Finally, users can choose the outputs of interest at the end of the file by the option of "yes" or "no"

  3. Run the code

    As all the preliminary preparations are right done, users can open the start window to activate the code:

    For Windows:

    xxxxxxxxxx
    Activate_inWindows.m

    For Linux:

    xxxxxxxxxx
    Activate_inLinux.m

    Next, add all the source code in compile path:

    xxxxxxxxxx
    addpath('.\iHydroSlide3D_src')

    Then start the simulation:

    xxxxxxxxxx
    prediction = Main_program('ControlFile.ProjectWindows')

7. Calibration

iHydroSlide3D v1.0 can be mainly evaluated upon hydrological and landslide levels. Streamflow observations from the local gauge stations are utilized for validation of the modeled discharge, along with which the statistical metrics such as Nash–Sutcliffe coefficient of efficiency (NSCE), Pearson correlation coefficient (CC), and relative bias are computed. Furthermore, more than a single gauge station is necessary when the very large scale or multi basins are involved. We also expect that the hydrologic process can be further calibrated by soil moisture data if the measurements are available.

For the landslide aspect, in-situ measurements (e.g., L, W, VL, and AL of failures) will be ideal documents for model validation and refinement. Such data not only serve for evaluation but provide more hints for the constraint of random procedure and model preparation. Point-like landslides and landslide scars are both acceptable for evaluation (computed by synthetic indices: LRclass and ROC−AUC).

8. Output

Users can control 2-D grid-based outputs by declaring in ControlFile (see as below, “yes” means output and “no” means do not output). The run time of the model depends on the number of outputs. A faster model runtime can be achieved by reducing the number of output variables. All outputs are spatially interpolated to the proper resolution and clipped to either the basic grids or the drainage area automatically. All the results are stored in the folder "Results".

xxxxxxxxxx
GOVar_Rain       =  no
GOVar_EPot       =  no
GOVar_EAct       =  no
GOVar_W          =  no
GOVar_SM         =  no
GOVar_R          =  no
GOVar_RS         =  no
GOVar_RI         =  no
GOVar_Infil      =  no
GOVar_FS         =  no
GOVar_tot_infil  =  no
GOVar_FS3D       =  no
GOVar_PF         =  no
GOVar_Volume     =  no
GOVar_Area       =  no
  • GOVar_Rain: the input precipitation; unit is mm/hour.
  • GOVar_EPot: the potential ET; unit is mm/hour.
  • GOVar_EAct: the actual ET; unit is mm/hour.
  • GOVar_W: the depth of water filling the pore space bucket "WM".
  • GOVar_SM: degree of soil moisture.
  • GOVar_R: the simulated discharge of each grid cell; unit is m³/s.
  • GOVar_RS: the depth of overland flow; unit is mm/hour.
  • GOVar_RI: the depth of interflow flow; unit is mm/hour.
  • GOVar_Infil: the infiltration value; unit is mm/hour.
  • GOVar_FS: the factor of safety calculated by infinite stability model.
  • GOVar_tot_infil: the accumulated infiltration value; unit is mm/hour.
  • GOVar_FS3D: the factor of safety calculated by 3D stability model.
  • GOVar_PF: the probability of landslide occurrence.
  • GOVar_Volume: the volume of landslide; unit is m3.
  • GOVar_Area: the surface area of landslide; unit is m2.

The model also provides results for the outlet location (see the table below), which are written to csv format series file. The model will automatically calculate the output pixel and output the csv file named "Outlet_Results". Users may also choose any specified output pixels so long as they are within the given watershed extent. For this purpose, users can define it in ControlFile. Here we choose the pixel location the same as basin outlet for example:

xxxxxxxxxx
NOutPixs         =  1
OutPixName1      =  Yuehe
OutPixLat1       =  32.7327747
OutPixLon1       =  108.7737
OutPixArea1      =  1200

As a consequence, the result file for the output pixel location will be named as "Outpix_Yuehe_Results". If, the observed streamflow is provided in OBS fold, three statistical metrics Nash–Sutcliffe coefficient of efficiency (NSCE), Pearson correlation coefficient (CC), and relative bias will be computed and stored in "Outpix_Yuehe_Results_Statistics". Please note that abnormalities in these factors may result if streamflow data are missing or all zeros, for which we would like to suggest a simulation including a complete flooding event.

Notes:

  • Complementary information such as measurements or observations will strongly affect the hydrogeological input and requires extreme care.

  • The prior information on parameters is necessary for this model and need to be handled with the utmost care. As a matter of fact, most of the parameters are determined by available datasets and field records, while few of them are controlled manually, based on computational experimental tests.

    Here we provide several data resources that have been tested and published:

    Harmonized World Soil Database v 1.2

    10.5194/hess-20-5035-2016

    10.1016/j.envsoft.2019.104607

    doi.org/10.1029/2018WR023539

    Users could find and extract necessary data from these available datasets according to their objective locations.

  • Users should very concerned about the "Landslide_density" whose value will significantly affect the output results and, even worse, a small value may yield meaningless results and unwanted consequences. Thus, the landslide density is necessary to be regularly tested when the code is applied for a fresh region. However, we still recommend a small value in the preliminary testing stage and gradually increase to get more mature results. Users may find more information in a relevant study in Mergili et al (2014).
