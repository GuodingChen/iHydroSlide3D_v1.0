iHydroSlide3D v1.0: Coupled hydrological model and three-dimensional landslide model

Version: 1.0 - 20210210 (10 February 2021)

 

1. iHydroSlide3D contains three core modules:

  • Hydromodel
  • 3D landslide model
  • Soil moisture downscaling method

2. Necessary data within folders

3. Detailed descriptions for files

The model features the feasibility of detached simulating resolution applied for submodules, by embedding a soil moisture downscaling method. Here we adopt adjectives "coarse" and "fine" denoting the resolution applied for hydrological module and landslide module, respectively.

  • DEM

    digital elevation model with coarse resolution.

  • FAC

    Flow accumulation with coarse resolution.

  • FDR

    Flow direction with coarse resolution.

  • LANDCOVER

    Land surface cover with coarse resolution.

  • MASK

    A matrix with binary value (0/NAN and 1) to delineate the objective river basin that serves as a extraction map.

  • SOIL

    Soil texture map with coarse resolution and is marked with USDA soil code from 1 to 12.

  • aspect_fine

    Geographical aspect angle (in degree) with fine resolution.

  • DEM_fine

    digital elevation model with fine resolution.

  • mask_fine

    mask map with fine resolution.

  • slope_fine

    Geographical slope angle (in degree) with fine resolution.

  • aspect_coarse

    Geographical aspect angle (in degree) with coarse resolution.

  • curvature_coarse

    Curvature with coarse resolution.

  • curvature_fine

    Curvature with fine resolution.

  • TWI_coarse

    Topographic wetness index with coarse resolution.

  • TWI_fine

    Topographic wetness index with fine resolution.

  • Precipitation data and Potential ET data

    Matrix format with coarse resolution.

  • COEM

    Overland runoff velocity coefficient.

  • FC

    Saturated hydraulic conductivity.

  • IM

    Percentage impervious area.

  • WM

    Available water capacity.

Notes: iHydroSlide3D v1.0 currently supports several different options for file formats (with ASCII, TIFF, and TXT) and map projections, of which the Geographic Tagged Image File Format (GeoTIFF) is preferred and recommended for its distinct advantage of including native compression capabilities, making the file sizes smaller.

4. Parameters

  • TimeStep

    Time step of the simulation.

  • W0

    Initial Value of Soil Moisture.

  • R

    Initial channel streamflow.

  • RS

    Initial unit overland streamflow.

  • SI0

    Initial value of Interflow reservoir.

  • RainFact

    The multiplier on the precipitation field.

  • Ksat

    The Soil saturate hydraulic conductivity.

  • WM

    The Mean Water Capacity.

  • B

    The exponent of the variable infiltration curve.

  • IM

    Impervious area ratio.

  • KE

    The factor to convert the PET to local actual.

  • coeM

    Overland runoff velocity coefficient.

  • Landslide_density

    Density of the random ellipsoid over the area.

  • Divide_tile_number

    Number of divisions of study area.

  • min_ae

    The minimum length of a random ellipsoid.

  • max_ae

    The maximum length of a random ellipsoid.

  • min_be

    The minimum width of a random ellipsoid.

  • max_be

    The maximum width of a random ellipsoid.

  • UserDefinedNcores

    Number of parallel computational cores. We also provide a option to identify the max number of cores in local environment, if the users state "yes" in "ControlFile".

  • A

    A parameter depending on the kind of soil and is linked to the peak shear stress at failure.

  • A numerical parameter which allows to translate the peak of the curve along the abscissa axis.

5. Software and hardware requirements

iHydroSlide3D v1.0 is written in Matlab and capable of running in a parallel manner, currently supporting Linux and Windows operating systems. Users are recommended to install the MATLAB platform with versions of R2018B and above in case that some unexpected errors will emerge due to the unrecognized functions. The code can currently run successfully on Windows 10 and CentOS release 6.6, but is expected to work in other LINUX and Windows environments, too. The computing core needs to be greater than or equal to 2 in a tested machine to ensure the computational efficiency. The best option is to submit the code to computing clusters or cloud computing services, however, PCs can also be attempted if the users could take their time.

6. Run the code step by step

  1. Prepare the necessary data

    All the data mentioned above should be acquired and preprocessed before the code being executed. The data must be correctly placed in the specified folders.

  2. Set the "ControlFile"

    Users may find the example of the case, for Windows:

    ControlFile.ProjectWindows

    and for Linux, please find:

    xxxxxxxxxx
    ControlFile.ProjectLinux

    Such file is so important that should be treated by users with utmost care. Users are required to set the simulation period by modifying "start time" and "end time". We note that there are two simulation durations including the main- and sub-durations. The former denotes the simulation of the hydrological processes, while the later controls the landslide model outputs. For instance, they can be set as:

    xxxxxxxxxx
    StartYear        =  2012 # in the format of yyyy
    StartMonth       =    06 # in the format of mm
    StartDay         =    05 # in the format of dd
    StartHour        =    03 # in the format of hh
    StartMinute      =    00 # in the format of uu
    StartSecond      =    00 # in the format of ss
    EndYear        =    2012 # in the format of yyyy
    EndMonth       =      06 # in the format of mm
    EndDay         =      05 # in the format of dd
    EndHour        =      14 # in the format of hh
    EndMinute      =      00 # in the format of uu
    EndSecond      =      00 # in the format of ss

    and for landslide model output, we can set:

    xxxxxxxxxx
    LandslideOutStart_data = 2012-06-05 03:00:00
    LandslideOutEnd_data = 2012-06-05 14:00:00

    The file contains all parameters above that need to be assigned with specific value. Part of them are provided a option to be read in a distributed manner, which can be easily modified by users by choosing "Uniform" and "Distributed" pattern. In this case, all the data is associated with the path and format of the file and, for example, is set as:

    xxxxxxxxxx
    ParamFormat      =  tif
    ParamPath        =  ".\Parameters\"

    Finally, users can choose the outputs of interest at the end of the file by the option of "yes" or "no"

  3. Run the code

    As all the preliminary preparations are right done, users can open the start window to activate the code:

    For Windows:

    xxxxxxxxxx
    Activate_inWindows.m

    For Linux:

    xxxxxxxxxx
    Activate_inLinux.m

    Next, add all the source code in compile path:

    xxxxxxxxxx
    addpath('.\iHydroSlide3D_src')

    Then start the simulation:

    xxxxxxxxxx
    prediction = Main_program('ControlFile.ProjectWindows')

7. Calibration

iHydroSlide3D v1.0 can be mainly evaluated upon hydrological and landslide levels. Streamflow observations from the local gauge stations are utilized for validation of the modeled discharge, along with which the statistical metrics such as NashSutcliffe coefficient of efficiency (NSCE), Pearson correlation coefficient (CC), and relative bias are computed. Furthermore, more than a single gauge station is necessary when the very large scale or multi basins are involved. We also expect that the hydrologic process can be further calibrated by soil moisture data if the measurements are available.

For the landslide aspect, in-situ measurements (e.g., , , , and of failures) will be ideal documents for model validation and refinement. Such data not only serve for evaluation but provide more hints for the constraint of random procedure and model preparation. Point-like landslides and landslide scars are both acceptable for evaluation (computed by synthetic indices: and ).

Notes:

  • Complementary information such as measurements or observations will strongly affect the hydrogeological input and requires extreme care.

  • The prior information on parameters is necessary for this model and need to be handled with the utmost care. As a matter of fact, most of the parameters are determined by available datasets and field records, while few of them are controlled manually, based on computational experimental tests.

    Here we provide several data resources that have been tested and published:

    Harmonized World Soil Database v 1.2

    10.5194/hess-20-5035-2016

    10.1016/j.envsoft.2019.104607

    doi.org/10.1029/2018WR023539

    Users could find and extract necessary data from these available datasets according to their objective locations.

  • Users should very concerned about the "Landslide_density" whose value will significantly affect the output results and, even worse, a small value may yield meaningless results and unwanted consequences. Thus, the landslide density is necessary to be regularly tested when the code is applied for a fresh region. However, we still recommend a small value in the preliminary testing stage and gradually increase to get more mature results. Users may find more information in a relevant study in Mergili et al (2014).
